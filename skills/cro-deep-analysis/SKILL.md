# CRO ディープ分析

GA4 データの数値と実際のページ構造を突き合わせ、「何を・どこを・どう変えるか」まで踏み込んだ改善提案を生成する。

**アンチパターン（禁止事項）:**
- 数値を整形し直しただけの「分析もどき」
- 「CTAを改善しましょう」「モバイルUXを向上させましょう」のような抽象的提案
- ページを見ずにデータだけで提案を書くこと

## 前提

- `analytics-mcp` MCP サーバーが接続済み
- 対象サイトの GA4 クリーンデータフィルタが確定済み（未確定なら先に `/ga4:clean-data` を実行）
- `skills/marketing-cro/SKILL.md` のフレームワークを判断基準として使用

## ワークフロー

### Phase 1: クリーンデータ取得（並行実行）

CLAUDE.md に確定済みフィルタがあればそれを使用。なければユーザーに確認。

以下の **8レポートを並行実行** する（全て過去30日、確定済み dimension_filter 適用）:

| # | dimensions | metrics | 目的 |
|---|-----------|---------|------|
| 1 | `deviceCategory`, `landingPagePlusQueryString` | sessions, bounceRate, averageSessionDuration, conversions | デバイス×ランディング別の直帰・滞在パターン |
| 2 | `sessionDefaultChannelGroup`, `eventName` | eventCount, totalUsers | チャネル別のコンバージョン行動差 |
| 3 | `deviceCategory`, `eventName` | eventCount, totalUsers | デバイス別のコンバージョン行動差 |
| 4 | `sessionSourceMedium` | sessions, totalUsers, bounceRate, averageSessionDuration, conversions | 流入元の質の比較 |
| 5 | `pagePath`, `eventName` | eventCount, totalUsers | ページ別のコンバージョンイベント発生状況 |
| 6 | `pagePath` | screenPageViews, totalUsers, averageSessionDuration, bounceRate | ページ別基本指標 |
| 7 | `deviceCategory` | sessions, totalUsers, bounceRate, averageSessionDuration | デバイス別基本指標 |
| 8 | `sessionDefaultChannelGroup` | sessions, totalUsers, bounceRate, averageSessionDuration, conversions | チャネル別基本指標 |

### Phase 2: クロス分析（データだけで発見を抽出）

Phase 1 の結果を**クロスで突き合わせ**、以下のパターンを特定する:

#### 必須チェック項目

1. **デバイス×ページの直帰率ギャップ**
   - モバイルとデスクトップで直帰率に15%以上の差があるページを特定
   - 「モバイル直帰率が高い」で終わらず、**どのページで**高いかを特定

2. **チャネル別コンバージョン効率**
   - チャネルごとのCVR（コンバージョン/セッション）を算出
   - 10倍以上の効率差があるチャネルペアを特定
   - トラフィック量は多いがCVR低いチャネル＝改善余地大

3. **コンバージョンのページ集中度**
   - コンバージョンイベントが発生しているページと発生していないページを分類
   - PVがあるのにCVイベント0件のページ＝CTA導線の死角

4. **ファネル断裂の特定**
   - `demo_complete` → `generate_lead` → `form_submit` 等、イベントの連鎖を追跡
   - デバイス別に追跡し、「モバイルでは途中で落ちる」等の断裂を発見

5. **高検討ユーザーの行動パターン**
   - 滞在時間が長い（上位20%）ページとそのユーザーの次の行動
   - 料金ページ等の高検討ページからコンタクトへの遷移率

6. **流入元の質のランキング**
   - 直帰率×滞在時間×CVR で流入元を質スコアでランク付け
   - 最高品質と最低品質の流入元を特定し、なぜ差があるかを考察

### Phase 3: 実ページ取得（並行実行）

Phase 2 で特定した問題に関連するページを **WebFetch で実際に取得** する。

最低限取得するページ:
- **トップページ**（ランディングの大半を占めるため）
- **コンタクト/お問い合わせページ**（コンバージョンの最終地点）
- **Phase 2 で問題が見つかったページ**（料金、特定のLP等）

取得時の注意:
- ページ構造（セクション数、セクション順序）を把握する
- CTA の位置・文言・数を記録する
- フォームがあればフィールド数・必須/任意・ボタン文言を記録
- モバイルファーストビュー（スクロールなしで見える範囲）を意識する

### Phase 4: データ×UX の突き合わせ

**ここが最重要。Phase 2 の数値発見と Phase 3 のページ構造を突き合わせる。**

各問題について以下のフォーマットで分析:

```
【問題】モバイル直帰率が /solutions で62%（デスクトップ28%）
【データ根拠】レポート#1: デバイス×LP の直帰率差
【ページ構造の原因】/solutions はデスクトップ3カラムレイアウト。
  モバイルでは縦積みになり、CTAがスクロール8回先。
  ファーストビューにはヘッダ画像しかない。
【具体的改善】
  - ファーストビュー内に「無料相談」ボタンを追加（44px以上）
  - 3カラム→タブ切替UIに変更し、スクロール量を1/3に削減
  - セクション2の見出し「ソリューション一覧」の下に
    「3分で最適なプランを診断→」のインラインCTA追加
```

### Phase 5: 改善提案の構造化

全ての改善提案を以下の形式でまとめる:

#### 各提案に必須の要素

| 要素 | 説明 |
|------|------|
| **データ根拠** | どのレポートのどの数値が根拠か（「モバイル直帰率42%」のように具体値） |
| **UX上の原因** | ページのどの構造が問題を引き起こしているか |
| **変更箇所** | 具体的にどのページのどの部分を変えるか |
| **変更内容** | 現状→変更後を明確に（ボタン文言、フィールド数、セクション順序 等） |
| **期待効果** | marketing-cro のベンチマークに基づく改善幅の見積もり |

#### ICE スコアで優先順位付け

全提案を ICE（Impact / Confidence / Ease）で 1-10 スコアリングし、テーブルで提示。

| 施策 | Impact | Confidence | Ease | ICE | 優先順位 |
|------|--------|------------|------|-----|----------|
| 例: フォーム3フィールド化 | 9 | 8 | 9 | 8.7 | 1位 |

#### チャネル戦略の提案

数値分析で判明したチャネル効率差に基づき、以下を提案:
- 低CVRチャネルの改善策（リンク先変更、投稿内容調整 等）
- 高品質チャネルの拡大策
- サブページのCTA導線追加

### Phase 6: 出力

最終的な出力は以下の構成:

```
## サマリー（3行以内）
全体の状況を端的に

## 重要発見（データクロス分析）
Phase 2 で見つかった重要なパターン（5-8個）
各発見に具体的な数値を添える

## 改善提案
Phase 5 の提案をICE順に
各提案にデータ根拠・UX原因・具体的変更・期待効果を含む

## チャネル戦略
流入元の質に基づく施策

## ICE優先度テーブル
全施策の一覧と優先順位

## 最優先アクション
今すぐやるべき1つの施策とその理由
```

### Phase 7: レポート生成（Word .docx）

Phase 6 のテキスト出力に加え、図表付きの Word レポートを自動生成する。

#### 手順

1. **venv 作成 + パッケージインストール**
   ```bash
   python3 -m venv /tmp/cro_report_env
   source /tmp/cro_report_env/bin/activate
   pip install python-docx matplotlib numpy
   ```

2. **データ構造化**: Phase 2 の分析結果を `REPORT_TEMPLATE.py` の `DATA` dict 形式に変換
   - 基本情報（サイト名、URL、GA4プロパティ、期間）
   - KPI（セッション、ユーザー、直帰率、CVR、推定損失）
   - 7グラフ用データ（損失リード、ファネル、CTAマトリクス、フォーム、チャネル、ICE、直帰率ギャップ）

3. **文章構造化**: Phase 4-6 の分析文章を `NARRATIVES` dict に変換
   - エグゼクティブサマリー、各セクションの分析テキスト、改善仕様、ロードマップ

4. **グラフ生成**: `CHART_SPEC.md` に従い7グラフを matplotlib で生成（180 DPI、Hiragino Sans）

5. **Word 出力**: `REPORT_TEMPLATE.py` の `generate_docx()` で8セクション + 表紙の .docx を生成

6. **出力先**: `{プロジェクトルート}/{サイト名}_CRO分析レポート.docx`

#### テンプレートファイル

- `skills/cro-deep-analysis/REPORT_TEMPLATE.py` — データ注入 → グラフ生成 → Word 出力の統合テンプレート
- `skills/cro-deep-analysis/CHART_SPEC.md` — 7グラフの仕様書（判断基準・算出ロジック・アノテーション）

#### 注意事項

- matplotlib の Hiragino Sans は Unicode 絵文字（✓ ⚠ 等）を描画できない。`[OK]` `[NG]` `[!]` で代替する
- フォントがない環境（Linux）では `font.family` を `Noto Sans CJK JP` に変更する
- グラフは `/tmp/cro_charts/` に PNG で一時保存し、Word に埋め込み後も残す（デバッグ用）
